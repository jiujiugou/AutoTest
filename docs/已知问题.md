# 已知问题与边界情况

## 已修复的问题（2026-05-09）

### 1. MonitorRepository 不持久化 IsTemplate
- **现象**：TEMPLATE 监控执行时报 "No execution engine found for target TEMPLATE"
- **根因**：仓储 INSERT/UPDATE/SELECT 都没处理 `IsTemplate` 和 `TemplateVariablesJson` 字段，从 DB 加载后 `IsTemplate` 恒为 `false`，管线跳过模板解析步骤
- **修复**：`MonitorEntity.IsTemplate` 改为计算属性 `=> Target?.Type == "TEMPLATE"`，不再依赖 DB 字段

### 2. DslRuntimeContext.CancellationToken 序列化崩溃
- **现象**：模板执行时 `RedisProgressStore.SaveAsync` 抛 `IntPtr` 序列化异常
- **根因**：`CancellationToken` 内部含 `IntPtr`，`System.Text.Json` 不支持
- **修复**：添加 `[JsonIgnore]`，该字段仅运行时使用，无需持久化

### 3. DI 注册导致所有 StepExecutor 拿到 TcpExecutionEngine
- **现象**：HTTP 模板步骤执行时报 "Expected TcpTarget"
- **根因**：4 个 `IExecutionEngine` 全部注册为 `AddScoped<IExecutionEngine, T>`，.NET DI 取单个时永远返回最后注册的 `TcpExecutionEngine`
- **修复**：注册具体类型 + 工厂方法，每个 `XxxStepExecutor` 注入正确的具体引擎

### 4. PythonStepExecutor 丢数据
- **现象**：Python 步骤执行成功后 Body 显示"Python 脚本执行成功"而非实际 stdout，ElapsedMs 永远为 0
- **修复**：从 `PythonExecutionResult` 读取 `StdOut` 和 `ElapsedMs`

### 5. HttpExecutionEngine MaxConcurrency=0 崩溃
- **现象**：`new SemaphoreSlim(0)` 抛 `ArgumentOutOfRangeException`
- **修复**：`MaxConcurrency <= 0 ? 1 : MaxConcurrency` 防御

### 6. PythonAssertion 无效操作符静默降级
- **现象**：输入 `"equl"`（拼写错误）被当作 `Equal`，不报错
- **修复**：无效操作符直接 `throw ArgumentException`

### 7. ExecutionEngine 顺序步骤和并行组不能交替
- **现象**：`[stepA, parallelB, stepC]` 实际执行 `[A, C, B并行]`
- **修复**：`StepSequence.Items` 统一有序列表，引擎遍历 `Items` 而非分别遍历 `Steps` 和 `ParallelGroups`

### 8. 模板步骤断言结果不持久化
- **现象**：DSL 内断言结果只影响 `IsTerminated`，不写入 AssertionResult 表
- **修复**：`StepExecutionRecord.Assertions` 记录每步断言 → Redis 持久化 → `RuntimeOrchestrationStep` 汇总到 `context.Result.Assertions`

### 9. 废弃的 src/OutBox/ 项目
- **问题**：旧的 OutBox 实现，在 .sln 中但无项目引用
- **修复**：从 .sln 移除，删除目录

## 现存边界情况

### 1. IsTemplate DB 列为废列
- 迁移 `AddMonitorTemplateFields` 建的列，仓储不读不写。保留无害，但占用空间
- 如需清理：新迁移 `Delete.Column("IsTemplate").FromTable("Monitor")`

### 2. DslSchemaValidator 不校验 parallel 字段
- `parallel` 的格式错误（缺失 mode、错误 timeout 格式）在运行时解析时才报错，而非创建时
- 影响：用户保存模板后执行才看到错误

### 3. 前端非 HTTP 监控没有断言编辑 UI
- `Task.vue` 只在 `targetType === 'HTTP'` 时显示断言表单
- TCP/DB/Python/Template 只能通过 API 直接配断言

### 4. 并行步骤变量冲突检测非线程安全
- 并行步骤各自在 localCtx 中检测冲突，合并时后来覆盖先来的
- 概率极低，前提是并行的两步恰好提取同名变量

### 5. 断路器作用域为单次执行
- `ExecutionEngine` 注册为 Scoped，每次执行新建实例
- 断路保护只在同一次执行的后续步骤有效，跨执行不共享

### 6. Outbox 无死信队列自动清理
- 超过最大重试次数的消息状态为 DeadLetter，无自动清理
- 需定期手动清理或加 TTL 策略

### 7. TcpAssertion 的 logger 永远为 null
- `TcpAssertionMap` 构建时不传 logger，`TcpAssertion` 构造函数 `logger = null`
- 影响：TCP 断言出问题时无日志

### 8. 监控执行幂等键需前端生成
- `IdempotencyKey` 由前端传入，重复提交可能产生重复执行记录
- 唯一过滤索引保护数据库层面不重复

## 性能注意事项

- **MonitorRepository.ListAsync**：一次查询 Monitor + 批量 IN 查询 Assertion，避免 N+1
- **CacheCommons**：TTL 随机抖动（0.9-1.1x）防惊群，空值缓存防穿透，SemaphoreSlim 防击穿
- **OutboxDispatcher**：5 并发处理，指数退避重试
- **AiWorker**：4 并发 + SemaphoreSlim，3 次指数退避
- **Hangfire**：WorkerCount=5，QueuePollInterval=1s
- **ExecutionRecord**：幂等键唯一过滤索引，心跳字段单独索引供看门狗查询
