# AutoTest 健康检查设计方案

## 概述

为 AutoTest 自动化监控平台设计生产级健康检查，支持 Docker 容器编排和 Kubernetes 就绪探测。遵循 ASP.NET Core 官方 `Microsoft.Extensions.Diagnostics.HealthChecks` 规范。

## 设计目标

| 端点 | 用途 | 检查范围 | Kubernetes 探针映射 |
| ---- | ---- | -------- | -------------------- |
| `GET /health` | 简单存活探测 | DB 连通性 | — |
| `GET /health/live` | 存活探针 | 进程是否存活（最小开销） | `livenessProbe` |
| `GET /health/ready` | 就绪探针 | 所有依赖是否可用 | `readinessProbe` |

- **live**：只检查进程不卡死，永远为健康（除非进程崩溃）。避免因外部依赖故障导致容器被无限重启。
- **ready**：检查 DB、Redis、Hangfire 等外部依赖，任一不可用即返回 503。用于将 Pod 从负载均衡中摘除。

## 2. Program.cs 注册

```csharp
// 在 builder.Services 注册阶段添加

using Microsoft.Extensions.Diagnostics.HealthChecks;

var cs = builder.Configuration.GetConnectionString("DefaultConnection")!;
var redisConn = builder.Configuration["Redis:ConnectionString"] ?? "localhost:6379";

builder.Services.AddHealthChecks()
    // 标签 "live" 仅包含轻量检查
    .AddCheck("self", () => HealthCheckResult.Healthy("process alive"), tags: new[] { "live" })

    // 标签 "ready" 包含所有外部依赖
    .AddCheck<WorkflowSchedulerHealthCheck>("workflow_scheduler", tags: new[] { "ready" })
    .AddCheck<PythonRuntimeHealthCheck>("python_runtime", tags: new[] { "ready" })
    .AddSqlServer(
        connectionString: cs,
        healthQuery: "SELECT 1",
        name: "sqlserver",
        failureStatus: HealthStatus.Unhealthy,
        tags: new[] { "ready" })
    .AddRedis(
        redisConnectionString: redisConn,
        name: "redis",
        failureStatus: HealthStatus.Unhealthy,
        tags: new[] { "ready" });
```

> `.AddCheck("self")` 是空检查，仅用于标记进程存活——即使所有外部依赖故障，`live` 仍返回 Healthy，防止数据库宕机导致容器被 Kubernetes 无限重启。

## 3. 端点映射（minimal API 风格）

放在 `app.UseEndpoints()` 和 `app.Run()` 之间的同一区域：

```csharp
app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = r => r.Tags.Contains("live")      // 仅执行标签 "live" 的检查
});

app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = _ => true                         // 执行全部检查
});
```

> 不使用 `AddRazorPages` / `MapHealthChecksUI`，保持最简依赖。

## 4. 返回 JSON 格式

```csharp
app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = _ => true,
    ResponseWriter = async (context, report) =>
    {
        var json = JsonSerializer.Serialize(new
        {
            status = report.Status.ToString(),
            duration = report.TotalDuration.TotalMilliseconds,
            entries = report.Entries.Select(e => new
            {
                name = e.Key,
                status = e.Value.Status.ToString(),
                duration = e.Value.Duration.TotalMilliseconds,
                description = e.Value.Description,
                error = e.Value.Exception?.Message
            })
        }, new JsonSerializerOptions { WriteIndented = false });

        context.Response.ContentType = "application/json";
        context.Response.StatusCode = report.Status == HealthStatus.Healthy ? 200 : 503;
        await context.Response.WriteAsync(json);
    }
});
```

## 5. 自定义 HealthCheck 实现

### 5.1 WorkflowScheduler 检查

验证 Hangfire 调度器是否响应（ping Hangfire 内部状态，执行一次轻量查询）：

```csharp
using Microsoft.Extensions.Diagnostics.HealthChecks;
using Hangfire;

public class WorkflowSchedulerHealthCheck : IHealthCheck
{
    private readonly IWorkflowScheduler _scheduler;   // 可选：直接检查 DI 能否解析
    // 或使用 Hangfire IMonitoringApi 检查队列状态

    public async Task<HealthCheckResult> CheckHealthAsync(HealthCheckContext ctx, CancellationToken ct)
    {
        try
        {
            var monitor = JobStorage.Current.GetMonitoringApi();
            var servers = monitor.Servers();
            if (servers.Count == 0)
                return HealthCheckResult.Unhealthy("No Hangfire servers running");

            return HealthCheckResult.Healthy($"Hangfire servers: {servers.Count}");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy("Hangfire unreachable", ex);
        }
    }
}
```

### 5.2 Python Runtime 检查

检测 `python` 命令是否可执行（`python --version`）：

```csharp
public class PythonRuntimeHealthCheck : IHealthCheck
{
    public async Task<HealthCheckResult> CheckHealthAsync(HealthCheckContext ctx, CancellationToken ct)
    {
        try
        {
            using var process = new System.Diagnostics.Process
            {
                StartInfo = new ProcessStartInfo
                {
                    FileName = "python",
                    Arguments = "--version",
                    RedirectStandardOutput = true,
                    RedirectStandardError = true,
                    UseShellExecute = false,
                    CreateNoWindow = true
                }
            };
            process.Start();
            await process.WaitForExitAsync(ct).WaitAsync(TimeSpan.FromSeconds(5));
            return process.ExitCode == 0
                ? HealthCheckResult.Healthy(await process.StandardOutput.ReadToEndAsync(ct))
                : HealthCheckResult.Unhealthy($"Exit code: {process.ExitCode}");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy("Python not available", ex);
        }
    }
}
```

## 6. Docker Compose 健康检查配置

```yaml
# api service
healthcheck:
  test: ["CMD-SHELL", "curl -sf http://localhost:80/health/live || exit 1"]
  interval: 15s
  timeout: 5s
  retries: 3
  start_period: 30s

# sqlserver service
healthcheck:
  test: /opt/mssql-tools18/bin/sqlcmd -S localhost -U sa -C -P "$${MSSQL_SA_PASSWORD}" -Q "SELECT 1" || exit 1
  interval: 10s
  timeout: 5s
  retries: 5
  start_period: 30s

# redis service
healthcheck:
  test: ["CMD", "redis-cli", "ping"]
  interval: 10s
  timeout: 5s
  retries: 5
```

> `docker-compose.yml` 中 `api` 的 healthcheck 应使用 `/health/live`，**不调用 `/health/ready`**。容器存活依赖 Redis/DB 上线时，用 depends_on 的 `condition: service_healthy` 处理，不是通过重启容器来解决。

---

## 7. 避免依赖故障导致容器无限重启

| 策略 | 说明 |
| ---- | ---- |
| `livenessProbe` 使用 `/health/live` | 永远不因外部依赖返回 503，Kubernetes 不会误杀容器 |
| `readinessProbe` 使用 `/health/ready` | 外部依赖故障时返回 503，Pod 从 Service Endpoints 摘除，但容器继续运行等待恢复 |
| `start_period: 30s` | 留出迁移和预热时间 |
| health check DB 查询用 `SELECT 1` | 最快响应，不产生锁争用 |
| Redis 连接失败不抛异常 | 使用 `ConnectionMultiplexer.IsConnected` 检查，不创建新连接 |

## 9. 与当前 `/health` 端点的关系

当前 `Program.cs` 中已有一个最小实现 `app.MapGet("/health", ...)` 做 DB 连通性检查。迁移方案：

- 保留 `/health` 用于简单手动探活
- 新增 `/health/live` 和 `/health/ready` 用于容器编排
- 两者的 DB 检查使用不同的实现路径（`/health` 直接用 ADO.NET，`/health/ready` 通过 `AspNetCore.HealthChecks.SqlServer`），避免单点故障误判
