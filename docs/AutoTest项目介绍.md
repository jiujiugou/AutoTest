# AutoTest — 自动化监控平台

> 一行命令部署，五分钟上线，失败钉钉通知 + AI 诊断。把"不知道服务挂了"变成"挂了 30 秒就知道"。

---

## 一句话说清楚

**你的 HTTP 接口、数据库、Redis、定时脚本、业务流程——定期检查，挂了钉钉通知你，AI 帮你分析为什么挂。**

---

## 你肯定遇到过

- 用户打电话说"系统打不开了"，你才知道 API 已经挂了半小时
- 证书过期，网站被浏览器标记不安全，客户截图来问
- 半夜数据库连接池满了，第二天早上才发现订单少了一半
- Redis 崩了但服务没报错，排查了两天才定位到

**这些问题都有一个共同特征：你不是第一个知道的人。**

AutoTest 做的事情很简单——**在你之前知道，并且告诉你为什么。**

---

## 它能监控什么

| 类型 | 我能监控 | 举个例子 |
|------|---------|---------|
| HTTP | API 接口是否正常 | 每 5 分钟请求 `/health`，返回非 200 就告警 |
| TCP | 端口是否可达 | Redis 6379、MySQL 3306 连不上就通知 |
| DB | 数据库数据是否正常 | 订单表行数突然变少、超时未处理的工单 |
| Python | 自定义脚本 | SSL 证书还有几天过期、定时任务是否跑完 |
| 模板 | 多步骤业务流程 | 登录 → 查订单 → 验金额，整条链路通不通 |

**五种类型，覆盖了你生产环境里 95% 的监控需求。**

---

## 它怎么通知你

监控失败后：

1. **钉钉机器人** 推送通知卡片——哪个服务、什么时候挂的、什么原因
2. **AI 自动分析** 执行日志，给出可能的根因（比如"连接超时，目标主机不可达"）
3. **仪表盘** 实时看到所有监控的状态——绿灯还是红灯

你不用盯着屏幕。有问题它找你，没问题你不需要打开它。

## 五分钟上手

```bash
# 1. 部署
cp .env.example .env   # 填两个配置项
docker compose up -d    # 一分钟启动

# 2. 打开浏览器
# 登录 → 任务调度 → 点「导入」→ 选「API 健康检查」→ 填你的 API 地址 → 确定

# 3. 配置钉钉
# .env 里填 Outbox__Webhook__Url = 你的钉钉机器人地址 → docker compose up -d
```

**从零到第一条监控在跑 + 钉钉能收到通知，五分钟。**

## 和市面上的东西有什么不同

| | Grafana / Prometheus | 蓝鲸 / Zabbix | AutoTest |
|---|---|---|---|
| 部署 | 需要集群 + 配置多 | 重量级 | 一行 docker compose |
| 监控对象 | 指标（CPU/内存/QPS） | 主机/网络 | 业务接口 + 数据 + 流程 |
| 使用门槛 | 需要学 PromQL | 需要培训 | 填 URL 点确定 |
| AI 诊断 | 无 | 无 | 失败自动分析 |
| 适用规模 | 大厂 / 有运维团队 | 中大型企业 | **小团队到中型团队** |

**AutoTest 不替代 Prometheus。它做的是 Prometheus 不做的——业务级别的黑盒监控。**

## 技术栈

后端 .NET 10 + Hangfire 调度 + SignalR 实时推送 + Dapper + Redis 锁 + Elasticsearch 日志

前端 Vue 3 + Element Plus

全部容器化：`docker compose up -d` 即启动 web、api、sqlserver、redis、elasticsearch 五个服务
