# 数据流详解

## 1. 监控创建流程

```
前端 Task.vue
  → TargetXxxForm.buildTargetConfig()           # 收集目标配置
  → MonitorsApi.create/post(dto)                 # POST /api/monitor
  → MonitorController.Create(dto)
  → MonitorService.AddAsync(dto)
      → ITargetMap.Map(dto.TargetConfig)         # JSON → HttpTarget/TcpTarget/...
      → MonitorEntity(id, name, target, ...)     # 创建实体
      → if IsTemplate: SetTemplateVariables(json)
      → IUnitOfWork.ExecuteAsync(tx =>
          → MonitorRepository.AddAsync(entity, tx)   # INSERT Monitor + Assertions
        )
  → 返回 monitorId
```

### MonitorDto → MonitorEntity 映射

```
MonitorDto.TargetType = "HTTP"  → HttpTargetMap  → HttpTarget
MonitorDto.TargetType = "TCP"   → TcpTargetMap   → TcpTarget
MonitorDto.TargetType = "DB"    → DbTargetMap    → DbTarget
MonitorDto.TargetType = "PYTHON"→ PythonTargetMap→ PythonTarget
MonitorDto.TargetType = "TEMPLATE"→ TemplateTargetMap → TemplateTarget(DslJson)
```

`MonitorEntity.IsTemplate` 是计算属性 `=> Target?.Type == "TEMPLATE"`，不需要持久化。

## 2. 监控执行流程（普通监控）

```
Hangfire 触发 WorkflowJob.ExecuteAsync(monitorId)
  → RedisLock.Acquire("dsl-run-{monitorId}")        # 防重复执行
  → MonitorRepository.GetByIdAsync(id)               # 加载 MonitorEntity
  → 幂等检查（IdempotencyKey）
  → ExecutionRecordRepository.TryAddRunningAsync()    # INSERT 执行记录
  → 心跳线程启动（每 30s 更新 HeartbeatAtUtc）
  ↓
Orchestrator.TryExecuteAsync(monitor, executionId, ...)
  → Pipeline.ExecuteAsync(context)
      ├── ExecutionStep.InvokeAsync
      │     → if IsTemplate: skip
      │     → ExecutionEngineResolver.Resolve(target.Type)
      │     → engine.ExecuteAsync(target)              # HttpExecutionEngine 等
      │     → context.Result = result
      ├── AssertionStep.InvokeAsync
      │     → AssertionEngine.EvaluateAsync(monitor.Assertions, result)
      │     → context.Result.Assertions = [...]
      ↓
  → 成功路径:
      → MonitorEntity.MarkSuccess()
      → UoW: UpdateAsync(monitor) + UpdateCompletionAsync(record) + AddAssertionResultsAsync
  → 失败路径:
      → MonitorEntity.MarkFailed()
      → BuildFailurePayload(monitor, result, ...)     # MonitorExecutionFailedPayload
      → BuildOutbox(payload)                          # OutboxMessage
      → UoW: UpdateAsync(monitor) + UpdateCompletionAsync(record) + outbox.AddAsync
  → 异常路径:
      → MonitorEntity.MarkFailed()
      → BuildFailurePayload(ex)
      → BuildOutbox(payload)
      → UoW: UpdateAsync(monitor) + UpdateCompletionAsync(record) + outbox.AddAsync
      → throw
  ↓
  → SignalR MonitorHub.monitorUpdated(monitorId, status)
  → 释放分布式锁
```

## 3. 模板监控执行流程（TEMPLATE）

模板监控的管线有额外两个步骤：

```
Pipeline 注册顺序:
  RuntimeOrchestrationStep  →  TemplateResolutionStep  →  AssertionStep  →  ExecutionStep

实际执行顺序（中间件模式，反向）:
  ExecutionStep.InvokeAsync:
    → if monitor.IsTemplate: next(); return;    # 跳过 —— 模板不直接执行引擎
  
  AssertionStep:
    → 运行 MonitorEntity.Assertions             # 顶层断言（通常为空）
  
  TemplateResolutionStep.InvokeAsync:
    → if !IsTemplate: next(); return;
    → 读取 TemplateVariablesJson → Dictionary<string,string>
    → DslParser.ParseAsync(Target.ToJson(), variables)
        → 正则匹配 {{var}} / {{var:default}}
        → 替换变量值
        → 构建 StepSequence (Items = Steps + ParallelGroups)
    → context.Items["DslPipelineContext"] = dslCtx
  
  RuntimeOrchestrationStep.InvokeAsync:
    → 获取 DslPipelineContext
    → ExecutionEngine.ExecuteAsync(dag, variables):
        ├── Redis 分布式锁
        ├── TryRestoreProgress (断点续跑)
        ├── for item in dag.Items:
        │     ├── StepDefinition → ExecuteSingleStep
        │     │     → VariableResolver.ReplaceJson(input, variables)
        │     │     → StepExecutorResolver.Resolve(step.Type)
        │     │     → executor.ExecuteAsync(input, ct)
        │     │     → 重试 + 退避
        │     │     → 变量提取 (JsonPath/Regex/Plain)
        │     │     → EvaluateAssertions → StepAssertionResult[]
        │     │     → 断路器记录
        │     ├── ParallelGroup → ExecuteParallelGroup
        │     │     → Task.WhenAll(子步骤)
        │     └── ProgressStore.SaveAsync(ctx) → Redis
        └── 返回 DslRuntimeContext
    → 汇总 completedSteps.Assertions → context.Result.Assertions
```

### DSL 步骤执行器映射

```
步骤 Type="http"   → HttpStepExecutor  → HttpExecutionEngine
步骤 Type="tcp"    → TcpStepExecutor   → TcpExecutionEngine
步骤 Type="db"     → DbStepExecutor    → DbExecutionEngine
步骤 Type="python" → PythonStepExecutor→ PythonExecutionEngine
```

每个 `XxxStepExecutor` 通过工厂方法注入正确的具体引擎类型，避免 DI 歧义。

### DSL 变量解析（两阶段）

```
阶段1 (DslParser):    {{baseUrl}}/api  +  variables["baseUrl"]="https://x.com"  →  https://x.com/api
阶段2 (VariableResolver):  运行时再次替换 input 中的 {{var}}（处理提取变量间的引用）
```

### DSL 断路保护

```
CircuitBreaker:
  - 键 = "{type}:{inputJson}"
  - 5 次失败 / 60s 窗口 → 打开
  - 60s 后 → 半开 → 一次成功即关闭
```

### DSL 进度持久化

```
RedisProgressStore:
  - 每步执行后 SaveAsync   → Redis SET "dsl:progress:{executionId}", JSON, TTL 6h
  - 启动时 RestoreAsync    → 从上次断点继续（CurrentStepIndex/CompletedSteps/Variables）
  - 完成时 CompleteAsync   → DEL key
```

## 4. AI 分析触发流程

```
Orchestrator 失败路径
  → BuildOutbox(payload)                     # OutboxMessage(Type="monitor.execution.failed")
  → 同一个 DB 事务内保存
  ↓
OutboxDispatcherHostedService (后台轮询)
  → DapperOutboxRepository.LockNextBatchAsync()  # 乐观锁领取
  → 反序列化 MonitorExecutionFailedPayload
  → MediatR.Publish(MonitorExecutionFailedEvent)
  ↓
AiAnalysisConsumer.Handle(event)
  → TargetSummaryBuilder.Build(TargetType, TargetConfig)    # 生成人类可读摘要
  → AiAnalysisInputDto(异常信息/堆栈/断言/追踪ID)
  → AiTaskService.EnqueueAsync(task)                        # INSERT AiTask
  ↓
AiWorker (后台轮询)
  → AiTaskService.TryTakeBatchAsync()          # 批量领取
  → Elasticsearch 查询 TraceId 相关日志（最近 20 条）
  → AiAnalysisPromptBuilder 构建 System + User Prompt
  → IAiClient.AnalyzeAsync(prompt)             # 调用 LLM
  → 反序列化 AiAnalysisOutputDto
  → AnalysisRepository.AddAsync(analysis)      # INSERT AIAnalysis
  → AiTaskService.MarkCompletedAsync(taskId)
```

### AI Prompt 结构

```
System: 你是自动化监控平台的故障分析专家...
User:
  - 监控目标摘要: "Template: 3 steps | Types: [http, http, tcp] | Parallel: 1 groups"
  - 错误快照: ExceptionType, ErrorMessage, StackTrace(截断2048)
  - 关键事件: Elasticsearch 中 ERROR/FATAL/fail/exception/timeout 的日志
  - 失败断言: 期望值 vs 实际值
  - 完整日志摘要: 最多 80 条，每条截断 120 字符
```

### AI 输出格式

```json
{
  "type": "TestFailure|ApiError|PerformanceIssue|SecurityIssue|Unknown",
  "severity": "low|medium|high|critical",
  "category": "自由文本分类",
  "summary": "一句话摘要",
  "rootCause": "根因分析",
  "suggestion": "建议操作",
  "impact": "single_request|module_level|system_level",
  "faultService": "故障归属服务",
  "confidence": 0.85,
  "errorChain": [
    { "type": "trigger|failure|consequence", "message": "..." }
  ]
}
```

失败回退（3 次重试后）返回 fallback JSON，category="AIError"。

## 5. 调度流程

```
HangfireWorkflowScheduler:
  RunNowAsync(monitorId)
    → BackgroundJob.Enqueue<WorkflowJob>(x => x.ExecuteAsync(monitorId, ...))
  
  UpsertDailyMonitorAsync(monitor)
    → RecurringJob.AddOrUpdate("monitor:{id}", x => x.ExecuteAsync(...), cron)
  
  每日自动执行:
    前端设置 AutoDailyEnabled + AutoDailyTime
    → Cron: "0 {HH} * * *" (北京时间)
    → MaxRuns 限制 → 达到上限自动关闭 AutoDailyEnabled
```

## 6. 陈旧执行看门狗

```
ExecutionWatchdogHostedService (每 60s)
  → SELECT * FROM ExecutionRecord WHERE Status = Running
  → 超时判定:
      HeartbeatAtUtc < Now - 2min  → 心跳丢失
      StartedAt < Now - 15min      → 总时长超限
  → UPDATE Status = Timeout
```
