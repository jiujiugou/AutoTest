# DSL 模板 vs 工作流 — 架构选择指南

> 版本: v1.0 | 状态: Draft

---

## 1. 为什么会有这个文档

在模板设计阶段，你问了一个关键问题：

> *"导入 JSON 里面会有多个任务，这多个任务就全部提交到库里，要是删除其中一个会有什么影响？"*

这个问题触及了**DSL 模板**和**工作流（Workflow）**两种架构的核心区别。这篇文档帮你理清它们的边界和选择依据。

---

## 2. 核心区别：一句话

| | DSL 模板（当前） | 工作流（Workflow） |
|---|---|---|
| **步骤是什么** | JSON 里的数组元素 | 独立的数据库记录 |
| **步骤能独立管理吗** | ❌ 不能 | ✅ 可以（独立 ID/状态/版本） |
| **步骤能单独运行吗** | ❌ 只能整个模板跑 | ✅ 可以单独触发 |
| **步骤间传参** | `{{变量}}` 自动传播 | 显式声明 inputs/dependsOn |
| **复杂度** | 低 | 高 |

---

## 3. DSL 模板的模型

```
┌─ MonitorEntity（数据库 1 条记录）────────────────────┐
│  id: "a1b2c3"                                         │
│  name: "用户注册流程"                                  │
│  targetType: "TEMPLATE"                                │
│  isTemplate: true                                      │
│  targetConfig: '{                                       │
│    "steps": [                    ← JSON 里的数组       │
│      { "name": "login", ... },   ← 不是独立数据库记录  │
│      { "name": "getProfile", ... },                     │
│      { "name": "verifyDB", ... }                        │
│    ]                                                    │
│  }'                                                     │
│  templateVariablesJson: '{ "host": "..." }'             │
└─────────────────────────────────────────────────────────┘
```

### 对修改的响应

| 操作 | 结果 | 影响范围 |
|---|---|---|
| **删除一个 step** | 修改 JSON 文本 | 仅影响下一次执行 |
| **新增一个 step** | 修改 JSON 文本 | 仅影响下一次执行 |
| **修改 step 参数** | 修改 JSON 文本 | 仅影响下一次执行 |
| **查看历史执行** | 查 ExecutionRecord 表 | 历史记录不受 JSON 修改影响 |

**关键**：steps 不是数据库对象，修改它们就是修改一段 JSON 文本，不会牵涉数据库表结构变化。

## 4. 工作流的模型（如果未来需要）

```
┌─ Workflow ──────────────────────────────────────────┐
│  id: "wf-001"                                       │
│  name: "用户注册流程"                                │
│                                                      │
│  ┌─ Task: login ──────────────────────────────┐     │
│  │  id: "t-001"                                │     │
│  │  type: HTTP                                 │     │
│  │  status: 启用                               │     │
│  │  独立 MonitorEntity 记录                    │     │
│  └─────────────────────────────────────────────┘     │
│           │                                          │
│  ┌─ Task: getProfile ────────────────────────┐     │
│  │  id: "t-002"                                │     │
│  │  dependsOn: ["t-001"]                       │     │
│  │  inputs: { "token": "{{t-001.token}}" }     │     │
│  └─────────────────────────────────────────────┘     │
│           │                                          │
│  ┌─ Task: verifyDB ──────────────────────────┐     │
│  │  id: "t-003"                                │     │
│  │  dependsOn: ["t-002"]                       │     │
│  └─────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────┘
```

### 对修改的响应

| 操作 | 结果 | 影响范围 |
|---|---|---|
| **删除一个 Task** | 删除独立数据库记录 | 需要处理下游依赖断裂 |
| **禁用一个 Task** | 设置 isEnabled=false | 依赖它的下游 Task 会失败 |
| **修改 Task 参数** | 更新独立记录 | 不影响其他 Task |
| **查看 Task 历史** | 每个 Task 有自己的执行记录 | 独立查询 |

## 5. 选择决策树

```
你的步骤需要独立管理吗？
│
├── 是 → 步骤有独立 ID？需要单独编历史版本？
│        可以被不同模板复用？
│        │
│        ├── 全部是 → 需要工作流架构
│        │
│        └── 部分需要 → 可以考虑混合模式
│                （主步骤用 DSL，关键步骤单独建监控）
│
└── 否（或不确定）→ 用 DSL 模板
        │
        ├── 步骤只是「验证流程」的一个环节
        ├── 步骤不会被人单独操作
        ├── 步骤数量 ≤ 20 个
        └── 想快速跑通，不想投入大量开发
```

## 6. 当前阶段：为什么 DSL 模板更合适

| 因素 | 说明 |
|---|---|
| **用户是谁** | 测试人员，关心"能不能跑通流程"，不是"步骤的独立生命周期" |
| **步骤数量** | 一个 JSON 通常 3-10 个步骤，不会膨胀到几百个 |
| **变量传递** | 当前 `extract` + `{{var}}` 模式已经满足 90% 场景 |
| **历史记录** | 每个模板有完整的 ExecutionRecord，可以看到每一步的结果 |
| **开发成本** | DSL 模板已经全部实现，工作流需要新的表、API、UI 编排引擎 |
| **试错成本** | DSL 模板用错了改 JSON 就行，工作流拆错了要重建任务关系 |

## 7. 如果未来想升级到工作流

DSL 模板到工作流**不是推倒重来**，底层数据结构已经做了预留：

```csharp
// 当前 StepSequence（已实现）
public class StepSequence
{
    public string Id { get; set; }                     // 预留
    public List<StepDefinition> Steps { get; set; }    // 串行步骤
    public List<ParallelGroup> ParallelGroups { get; set; } // 并行组
}

// 扩展为 DAG 时只需加一个字段（已预留扩展点）
public class StepDefinition
{
    public string Name { get; set; }
    public string Type { get; set; }
    public JsonElement Input { get; set; }
    public List<string> DependsOn { get; set; }         // ← 新增：依赖声明
    public List<ValueExtractor>? Extract { get; set; }
    // ...
}
```

**升级路径**：

```
阶段 1（当前）                阶段 2（可選）                   阶段 3（未来）
┌──────────────┐          ┌──────────────────┐          ┌──────────────────────┐
│ DSL 模板     │          │ 模板 + 独立步骤    │          │ 完整工作流            │
│              │          │                   │          │                       │
│ JSON 定义    │   ──►    │ JSON 定义         │   ──►   │ 可视化拖拽            │
│ one-step 执行│          │ + 关键步骤可单独建 │          │ DAG 调度              │
│ 变量自动传播 │          │ + 断言可提取到模板 │          │ 步骤版本管理          │
└──────────────┘          └──────────────────┘          └───────────────────────┘
```

## 8. 常见问题

### Q1：JSON 里删了一个 step，之前的执行记录还能看吗？

能。执行记录存储在 `ExecutionRecord` 表，`resultJson` 字段记录了每一步的结果。JSON 修改不影响已有的历史记录。

### Q2：一个模板跑一半失败了，能重跑吗？

当前设计是**整体重跑**。如果 step 1 成功、step 2 失败、step 3 未执行，修复后重跑会重新执行所有步骤。
未来可以加 `resumeFrom` 参数实现断点续跑。

### Q3：我的步骤 A 需要上一步的响应里的 token，怎么做？

用 `extract` 提取：

```json
{
  "name": "login",
  "type": "http",
  "input": { "url": "{{host}}/api/login", "method": "Post" },
  "extract": [
    { "name": "token", "source": "Body", "method": "JsonPath", "expression": "$.data.token" }
  ]
}
// → 生成变量 {{token}} 和 {{login.token}}
// 后面的步骤: {{login.token}}
```

### Q4：模板里的步骤能单独点"运行"吗？

不能。步骤不是独立数据库记录，没有独立 ID。模板只能整体运行。
如果你需要单独运行某个步骤，把它拆成一个单独的 HTTP/TCP/Python 监控任务。
